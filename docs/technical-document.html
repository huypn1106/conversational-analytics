Executive Summary

Stats Center is an internal conversational analytics platform that allows users to query business data using plain English. The system automatically translates natural-language questions into SQL, executes them against a StarRocks analytical database, and returns interactive Plotly charts — all in real-time via Server-Sent Events (SSE).

The platform eliminates the SQL knowledge barrier for data exploration, enables iterative follow-up queries using session context, and surfaces AI-generated insights alongside every chart — making analytics accessible to every business stakeholder.

Core Value Proposition: Ask a question in plain English → get an interactive chart and natural-language insight in under 10 seconds, with no SQL knowledge required.

End-to-End Request Flow

The following sequence describes what happens from the moment the user submits a question until the chart renders.

// Step-by-step flow for a new query

1. User types question → React calls streamChat(message, session_id)
2. Frontend: POST /api/chat  { message, session_id }
3. Backend:  Resolve or create Redis session
4. Backend:  SemanticRouter.classify(message) via LLM
            → returns: "chart_query" | "follow_up" | "clarification" | "export"
5. SSE event emitted: { type:"status", data:"Processing your question..." }
6. VannaAgent.load_schema()  → SHOW CREATE TABLE * from StarRocks (cached)
7. VannaAgent.generate_sql() → LLM call with schema DDL + history context
8. SSE event: { type:"sql", data:"SELECT ..." }
9. VannaAgent.execute_sql()  → pymysql → StarRocks → pandas DataFrame
10. SSE event: { type:"table", data:{ columns, rows } }
11. VannaAgent.generate_chart() → LLM infers best chart type → Plotly JSON
12. SSE event: { type:"plotly_chart", data:{ data:[], layout:{} } }
13. VannaAgent.generate_summary() → LLM writes 2-3 sentence insight
14. SSE event: { type:"summary", data:"..." }
15. RedisSessionManager.update_last_result(sql, result, chart)
16. SSE event: { type:"done", data:null }
17. Frontend: Plotly chart renders, SQL panel updates, summary displays

Key Design Decisions

Singleton Agent via DI: VannaAgent is instantiated once at startup via FastAPI's dependency injection. This prevents re-loading the schema DDL on every request and keeps LLM clients alive across the process lifetime.
Thread-safe SQL Execution: StarRocksRunner.run_sql() is synchronous (pymysql). It is wrapped with asyncio.to_thread() to prevent blocking the async event loop. A new pymysql connection is created and closed per request — no connection pooling yet.

The VannaAgent class in core/vanna_agent.py orchestrates four sequential LLM calls per query.

1. Schema Loadload_schema()StarRocks SHOW CREATE TABLEDDL string (cached in memory)

5. Summarygenerate_summary()question + top-10 rows preview2–3 sentence natural-language insight

Follow-up Query Handling

When the SemanticRouter classifies intent as follow_up, the agent injects a system message containing the previous SQL before calling the LLM. This allows the model to intelligently modify (e.g., change filters, add dimensions) or reuse (e.g., chart type changes) the prior query.

if intent == "follow_up" and session.last_sql:
    system_msg = (
        f"Previous SQL:\n```sql\n{session.last_sql}\n```\n"
        "If user wants to change chart type only, output EXACT same SQL."
    )
    history.append({"role": "system", "content": system_msg})

LLM Service

OllamaLlmService uses the OpenAI Python SDK pointed at Ollama's OpenAI-compatible endpoint (http://localhost:11434/v1). This means the LLM provider can be swapped to any OpenAI-compatible API (OpenAI, Azure, vLLM) by changing a single env var.

StarRocks System Prompt

A dedicated system prompt is used for SQL generation to constrain model behavior:

  • Generate valid StarRocks SQL (MySQL-compatible dialect)
  • Use schema table/column names exactly
  • Always use column aliases; always include GROUP BY for aggregations
  • Limit to 1000 rows unless otherwise specified
  • Prefer APPROX_COUNT_DISTINCT over COUNT(DISTINCT ...) for large datasets
  • Return only the SQL — no markdown, no explanation

Folder Management

Folders are stored as a Redis Hash at folders:all. Sessions reference their folder via folder_id. When a folder is deleted:

  • delete_conversations=true: All sessions in folder are deleted from Redis.
  • delete_conversations=false (default): Sessions are moved to root (folder_id → None).

Semantic Router

Before invoking the full Vanna pipeline, every user message is classified into one of four intents:

IntentTriggerAction
chart_queryNew, unrelated data questionFull NL→SQL→Chart pipeline
follow_upRefinement of previous resultPipeline with previous SQL injected into context
clarificationToo short or ambiguousReturn clarification prompt immediately (no SQL)
exportContains "export", "csv", "download"Return placeholder message (Phase 3)

Classification uses an LLM call with a tight prompt and max_tokens=10. If the LLM call fails, a keyword-based heuristic fallback is used (checking for words like "change", "also", "what about", "make it a pie chart", etc.).

SSE Event Schema

The /api/chat endpoint streams text/event-stream with typed JSON payloads. Each line is:

data: {"type":"<event_type>","data":<payload>}
status
string — Progress message. First event includes session_id. E.g. "Analyzing database schema..."
sql
string — The generated SQL query. Rendered in the SQL viewer panel.
table
{ columns: string[], rows: any[][] } — Raw query results for the data table component.
plotly_chart
{ data: [], layout: {} } — Full Plotly figure spec. Passed directly to <Plot />.
summary
string — 2–3 sentence natural-language insight from the LLM.
error
{ message: string, code: string } — Pipeline error. Displayed inline. Always followed by done.
done
null — Stream terminator. Frontend closes the reader and re-enables the input.
debug
{ step, messages, system_prompt? } — LLM prompt debug info. Visible in the Debug panel (internal use).

Frontend Architecture

The frontend is a Vite + React 19 SPA. It communicates with the backend exclusively through the SSE chat endpoint and the REST sessions/folders API.

State Management

There is no global state library (Redux, Zustand). All chat state is managed in the useChat custom hook, which is owned by AppContent and passed to child components as props.

// hooks/useChat.js — exported state & actions
const {
  messages,     // Message[] — rendered chat bubbles
  isStreaming,  // bool — disables input field
  sessionId,    // string | null
  sendMessage,  // (text) => void — triggers SSE stream
  cancelStream, // () => void — AbortController.abort()
  clearChat,    // () => void — reset messages
  loadSession,  // (id) => void — load history from API
  createSession // () => void — POST /api/sessions
} = useChat();

Phase Roadmap

✅ Phase 1 — Core Analytics Pipeline (Complete)

NL→SQL→Chart pipeline via SSE. Redis session state. Semantic intent routing. Follow-up query context. Chat history with folders, drag-and-drop, cascading delete.

🔄 Phase 2 — Enhanced AI & Schema Intelligence

Full Vanna agent integration with vector embeddings for schema retrieval. Training pipeline on query examples. Automatic table JOIN suggestions for multi-table schemas.

⏳ Phase 3 — Export, Auth & Multi-tenancy

CSV/JSON data export. User authentication (OAuth2/JWT). Per-user session isolation. Rate limiting enforcement. Custom chart theming.

⏳ Phase 4 — Scheduled Reports & Collaboration

Saved queries / dashboards. Scheduled report delivery via email. Shared chart links. Annotation and comment threads on charts.

Risks & Mitigations

RiskSeverityMitigation
LLM generates invalid SQLHighFallback chart on JSON parse error. Try/catch on SQL execution returns error SSE event. User can rephrase or follow-up.
StarRocks query timeoutMedium120s read timeout on pymysql connection. LIMIT 1000 enforced in system prompt. Future: query timeout header.
Redis session expiry mid-conversationMediumTTL refreshed on every access. If session is missing, backend auto-creates a new one. UI shows new session badge.
Ollama LLM unavailableHighReturns error SSE event with clear message. Health endpoint reports LLM status. Future: retry with exponential backoff.
LLM hallucinating column namesMediumFull DDL injected into every request. StarRocks returns MySQLError on invalid columns — caught and returned as error event.
No input sanitization on LLM promptsLow (internal only)Prompt injection risk is low for internal use. For external deployment, input length is capped at 2000 chars.
